# 프리셋 하네스 환경 구축
[!important] 프로젝트 진행에 있어 충분한 상의와 의논을 통해 작은 단위로 진행해야 함.
갑자기 모든 환경을 한 번에 만들어버리면 안됨.

## 개요
- 여러 client의 다양한 요구사항에 대응할 수 있는 일관된 틀을 가진 하네스 환경
- 기능 및 디자인 요구사항, 개발 스펙은 client 가 FDE(Forward Deployed Engineer)와의 커뮤니케이션을 통해 확립
- 개발 스펙에 따라 선택 적용할 수 있는 컨벤션, 설정에 대한 프리셋 구축
- client의 디자인 요구사항이 없거나 부족할 때, 제안할 수 있는 약 2~3개의 프리셋 구축

### 1. 요구사항 분석
- client와 FDE의 회의를 통해 작성해야하는 최적의 요구사항 form을 미리 정의
- 하나의 문서가 아닌, 여러 방면의 요구사항 문서를 작성하도록 해도 괜찮다. (ex_개요, 디자인, front, back 나눠서 받는 형태)

### 2. 화면 설계 및 작업 (frontend)
- front 개발 스펙에 따라 선택 사용할 프리셋 준비 필요 (React, Typescript, Next, Android, IOS 등)
- 코드 일관성 및 공통 컴포넌트 활용, 코드 퀄리티를 위한 컨벤션 정의 필요

### 3. 서버 환경 개발 (backend)
- backend 개발 스펙에 따라 선택 사용할 프리셋 준비 필요 (Springboot, Node.js, Python 등)
- 유지보수를 위한 코드 일관성, 가독성을 위한 컨벤션 정의 필요

### 4. DB 구축 (database)
- 선택된 DB 종류에 따른 테이블, 컬럼 등 정의 시 주의사항 명시
- 네이밍 관련 컨벤션 지정

### 5. 전체적인 요구사항
- 전체 플로우에 대한 고민이 필요하다.
- 예를 들어, 분석 > 설계 > 개발 > 검증 > 테스트 와 같은 플로우 진행 시 각각의 단계를 트리거할 수 있는 command와 모든 플로우를 한 번에 실행하도록 하는 command를 두었으면 좋겠다.
- 분석 > 설계 흐름에서 누락되거나 충분하지 않은 내용, 또는 개발 > 검증(테스트) 흐름에서 실패한 테스트 케이스가 있는 경우, 와 같이 루프로 돌면서 반복 검증해야한 경우 잘 유의해서 설계가 필요하다.
- 컨텍스트 유지와 성능을 위해 여러 agent로 작업을 나누어 진행, 혹은 병렬로 진행하는 방향을 적극적으로 고려했으면 좋겠다.
- 모든 과정에서 히스토리를 추적할 수 있는 git commit 전략을 가져갔으면 좋겠다.
- 전체적으로 프로젝트를 진두지휘하는 orchestrator 역할의 에이전트를 중심으로 여러 역할을 담당하는 에이전트들에게 작업을 위임하는 형식이 되었으면 좋겠다.
- 각 에이전트들은 서로 소통을 위해, 기록을 위해 문서 작성을 꼼꼼히 한다.
- 특히 요구사항 분석 후 작성되는 설계서는 추후 유지보수 및 기능 확장, 수정의 근거와 자료로 사용될 예정이다.
- claude code 환경에서 활용할 수 있는 여러 장치(skills, hooks, commands, rules 등)들을 적극적으로 활용해야한다.
- 이미 많이 사용되고 있는 범용 하네스 환경 분석 자료를 통한 벤치마킹을 진행할 예정이다.
